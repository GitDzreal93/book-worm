# Book-Worm 项目宪法

## 核心原则

### 一、代码质量

所有源代码必须遵守以下不可协商的质量标准：

- 必须启用 TypeScript 严格模式。禁止使用 `any` 类型，仅在边界代码（如第三方
  库适配器）中允许，且必须附带 `// SAFETY:` 注释说明理由。
- `eslint.config.mjs` 中定义的 ESLint 规则必须以零警告通过。规则可以收紧，
  但未经宪法修订不得放宽。
- 每个模块必须具有单一、明确的职责。模块间禁止循环依赖。
- 共享工具函数和类型必须放置在指定目录（如 `src/lib/`、`src/types/`）中。
  禁止跨模块复制逻辑——当一个模式在两个及以上文件中出现时，必须立即提取为
  共享代码。
- 所有公共函数和导出组件必须包含 JSDoc 或 TSDoc 注释，说明用途、参数和
  返回值。

**依据**：一致且结构良好的代码能降低上手成本、防止回归，并支持安全的并行开发。

### 二、测试标准

测试纪律贯穿开发全流程：

- 所有新功能必须包含单元测试。最低覆盖率阈值：新增代码行数的 80% 必须被
  测试覆盖。
- 关键用户路径（认证、数据持久化、支付流程、核心阅读功能）必须具备集成测试，
  覆盖完整的请求生命周期。
- 测试必须是确定性的——不得依赖系统时钟、随机排序或外部服务。使用可控的种子
  mock 和测试夹具。
- 测试文件必须镜像源码目录结构：`src/foo/bar.ts` 对应的测试文件必须位于
  `__tests__/foo/bar.test.ts` 或相邻的 `bar.test.ts`。
- 所有测试必须在 CI 中通过后方可合并。测试套件失败是发布阻断项。

**依据**：自动化测试是防止回归的首要安全网，也是在重构过程中验证正确性的最快
方式。

### 三、用户体验一致性

每个用户可见的界面必须保持连贯、可预期的体验：

- 统一的共享组件库是 UI 元素的唯一真实来源。禁止创建与组件库功能重复的临时
  或一次性组件。
- 设计令牌（颜色、间距、排版、圆角等）必须在唯一规范位置定义，并在整个应用
  中消费。令牌定义文件之外禁止硬编码样式值。
- 响应式布局必须在三个断点下验证：移动端（< 640px）、平板端（640–1024px）
  和桌面端（> 1024px）。
- 错误状态、加载状态和空状态必须为每个面向用户的功能明确设计和实现。缺少
  这三种状态的功能视为未完成。
- 无障碍访问必须达到 WCAG 2.1 AA 标准：语义化 HTML、键盘可导航性、足够的
  颜色对比度和适当的 ARIA 属性。

**依据**：用户基于一致的模式建立心智模型。不一致会侵蚀信任并增加认知负荷。

### 四、性能要求

性能预算被视为功能性需求，而非愿景目标：

- 服务端渲染页面的最大内容绘制（LCP）在模拟 4G 网络下不得超过 2.5 秒。
- 首页加载的客户端 JavaScript 包体积不得超过 200 KB（gzip 压缩后）。超过
  50 KB（gzip 压缩后）的路由必须实施路由级代码分割。
- 数据库查询必须在所有过滤和关联字段上使用索引列。禁止 N+1 查询——使用
  预加载或 DataLoader 模式解决关联查询。
- 图片必须以现代格式（WebP/AVIF）提供，并附带响应式 `srcset` 属性。禁止
  上传未经优化的位图。
- API 响应时间在 p95 百分位下，读操作不得超过 500 ms，写操作在正常负载下
  不得超过 1 秒。

**依据**：性能直接影响用户留存率。预算机制确保性能退化在部署前被捕获，而非
上线后才发现。

## 技术栈

- **运行时**：Next.js（App Router）+ TypeScript 严格模式
- **数据库**：Prisma ORM，迁移通过 `prisma migrate` 管理
- **样式**：Tailwind CSS，基于项目共享设计令牌系统
- **代码检查**：ESLint，配置文件为 `eslint.config.mjs`
- **测试**：Vitest（单元测试）、Playwright（集成与端到端测试）

所有技术选型锁定主版本号。升级主版本依赖需要创建专用特性分支、更新测试，
并提交宪法修订记录版本变更和迁移说明。

## 开发工作流

- 所有工作必须通过特性分支进行，分支命名格式为 `###-feature-name`，源自
  speckit 特性追踪系统。
- 拉取请求必须引用特性规格文档，且必须在评审前通过所有 CI 检查（lint、
  类型检查、测试）。
- 代码评审必须验证是否符合上述四项核心原则。评审者可因任何原则违规阻止合并。
- 合并至 `main` 分支的提交必须触发生产部署就绪检查。

## 治理规则

本宪法是项目标准的权威来源。它优先于所有临时约定、口头协议和未记录的实践。

- **修订**要求：(1) 书面提案描述变更内容及理由，(2) 项目维护者批准，
  (3) 按照以下版本策略进行版本号升级。
- **版本策略**：
  - MAJOR（X.0.0）：原则删除、重新定义或不向后兼容的治理变更。
  - MINOR（0.X.0）：新增原则或对现有指导进行实质性扩展。
  - PATCH（0.0.X）：措辞澄清、修正、非语义性质的改进。
- **合规审查**：每个特性规格和实施计划在开发开始前必须通过宪法检查门控。
  不合规的计划将被退回修订。
- **运行时指导**：参见 `CLAUDE.md` 和 `AGENTS.md` 获取开发会话中各项原则
  的执行细则。
